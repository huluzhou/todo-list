# 优先级管理功能 - 设计文档

> 设计日期：2025-02-24

## 1. 目标

- 为待办增加**优先级**管理：三档含义明确（紧急重要 / 重要不紧急 / 一般）。
- **未完成任务**按优先级排序；**同组内**旧在上、新在下。
- **添加待办时**选择优先等级；**列表中**可修改任意一条的优先级，改后重排并保存。
- 现有 `todos.json` 无 `priority` 时视为「一般」，更新应用后可设定并生效。

## 2. 优先级档位（方案三）

| 取值 (key)         | 展示文案   | 含义           |
|--------------------|------------|----------------|
| `urgent_important` | 紧急重要   | 又急又重要，优先做 |
| `important`       | 重要不紧急 | 重要但可排期   |
| `normal`          | 一般       | 普通事项       |

## 3. 数据模型与持久化

- **Todo 增加字段**：`priority: String`，取值上述三种 key。
- **存储**：仍在现有 `todos.json`，与 `id / text / done / order` 一并读写。
- **兼容**：读取时缺 `priority` 视为 `normal`，不要求迁移旧文件。
- **后端**：Rust `Todo` 增加 `priority: String`；`TodoRaw` 中 `priority` 为 `Option<String>`，缺省补 `"normal"`。

## 4. 排序规则

- **整体顺序**：先按 **完成状态**（未完成在前、已完成在后），再在 **未完成** 内按 **优先级**（紧急重要 → 重要不紧急 → 一般），同优先级内按 **order 升序**；**已完成**区块内仅按 order，不再按优先级区分。
- **同组内**：按 `order` 升序，即**旧的在上、新的在下**（先添加的在上，后添加的在下）。
- **与现有逻辑**：在现有「先 done 再 order」基础上，仅在未完成段插入按 priority 的排序；勾选完成/取消完成时的「移底 / 恢复最上」及 order 重算逻辑保持不变，重排时按新顺序（含 priority）赋 order。

## 5. 添加待办

- 在现有「文案 + 添加按钮」旁增加**优先级下拉**，三档：紧急重要 / 重要不紧急 / 一般。
- 默认选中「一般」；提交时为新待办写入对应 `priority`，`order` 仍为当前列表长度（新在队尾）。
- 表单校验仍仅校验文案非空。

## 6. 列表展示与修改优先级

- 每条待办展示**优先级文案**（或下拉）：与文案、复选框、删除按钮同行。
- **修改**：每行提供优先级下拉，与添加时一致；选择后更新该项 `priority`，按新规则重算显示顺序并重赋 `order`，调用 `save_todos` 写回并重渲染。

## 7. 边界与兼容

- 空列表、仅一条、全部完成/未完成：排序与重赋 order 逻辑均适用，无需特殊分支。
- 旧数据无 `priority`：视为 `normal`，更新应用后可设定并保存，此后即带 `priority`。

## 8. 实现位置

- **后端**：`desktop-todolist/src-tauri/src/storage.rs` 扩展 `Todo` / `TodoRaw`，`load_todos` 补全缺省 `priority`。
- **前端**：`desktop-todolist/src/main.js` 与 `index.html` / `styles.css`：排序逻辑、添加表单下拉、列表项展示与下拉修改、`handleChangePriority` 等；`save_todos` 接口不变，仅 payload 中 todo 带 `priority`。
