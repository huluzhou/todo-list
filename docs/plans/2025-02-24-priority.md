# 优先级管理功能 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 为待办增加三档优先级（紧急重要 / 重要不紧急 / 一般），未完成任务按优先级排序、同组内旧上新下；添加时选择优先级，列表中可修改优先级并重排保存；旧数据无 priority 时视为「一般」。

**Architecture:** 后端 Todo 增加 `priority` 字段，反序列化缺省为 `"normal"`；前端排序为 done → priority（紧急重要→重要不紧急→一般）→ order 升序；添加表单与列表项均用下拉选择/修改优先级，修改后按新顺序重赋 order 并 save_todos。

**Tech Stack:** Tauri 2, Vanilla JS, Rust (serde), 设计见 `docs/plans/2025-02-24-priority-design.md`

### Task 1: 后端 - Todo 增加 priority 字段并兼容旧数据

**Files:**
- Modify: `desktop-todolist/src-tauri/src/storage.rs`（Todo、TodoRaw、load_todos）

**Step 1: 编写失败测试（缺 priority 的 JSON 反序列化后得到 "normal"）**

在 `storage.rs` 的 `#[cfg(test)] mod tests` 中增加：反序列化不含 `priority` 的 JSON 到内部结构后，经默认逻辑得到 priority 为 `"normal"`。因 `TodoRaw` 为私有，测试通过公开行为验证：先增加 `pub(crate) fn default_priority() -> &'static str { "normal" }`（或 `pub const DEFAULT_PRIORITY: &str = "normal"`），在 tests 里 `assert_eq!(default_priority(), "normal");`。随后在 Step 2 中实现 Todo/TodoRaw 的 priority 与 load 补全时使用该默认值。

**Step 2: 实现 Todo / TodoRaw 的 priority 与 load 补全**

- `Todo` 增加 `pub priority: String`。
- `TodoRaw` 增加 `#[serde(default)] priority: Option<String>`。
- 在模块内定义 `const DEFAULT_PRIORITY: &str = "normal";`（或 `pub(crate) fn default_priority()`）。
- `load_todos` 映射时：`priority: r.priority.unwrap_or_else(|| DEFAULT_PRIORITY.to_string())`。
- `save_todos` 会自然序列化 `priority`，无需改。

**Step 3: 运行测试**

```bash

```

预期：现有 2 个测试通过，新增默认值测试通过。

**Step 4: Commit**

```bash

git commit -m "feat(storage): Todo 增加 priority 字段，缺省为 normal"
```

### Task 2: 前端 - 排序规则加入优先级（未完成段）

**Files:**
- Modify: `desktop-todolist/src/main.js`

**Step 1: 定义优先级顺序常量与排序**

在 `main.js` 顶部（在 `let todos = [];` 附近）增加：

```javascript
/** 优先级 key 的显示顺序（未完成区内）：紧急重要 > 重要不紧急 > 一般 */
const PRIORITY_ORDER = { urgent_important: 0, important: 1, normal: 2 };
function priorityRank(p) {
  return PRIORITY_ORDER[p] ?? PRIORITY_ORDER.normal;
}
```

修改 `sortedTodos(list)`：先按 `done`（未完成在前），再按 `priorityRank(a.priority) - priorityRank(b.priority)`，再按 `(a.order ?? 0) - (b.order ?? 0)`。

**Step 2: 运行应用与现有行为**

```bash
cd desktop-todolist && npm run tauri dev
```

确认列表仍按未完成/已完成分段；此时尚无 priority 数据，全部按 normal 处理，顺序应与原先一致（同组内按 order）。

**Step 3: Commit**

```bash
git add desktop-todolist/src/main.js
git commit -m "feat(ui): 未完成区内按优先级排序"
```

### Task 4: 列表中展示优先级并可下拉修改

**Files:**
- Modify: `desktop-todolist/src/main.js`
- Modify: `desktop-todolist/src/styles.css`（可选，下拉样式）

**Step 1: renderTodos 中每项增加优先级下拉**

在 `renderTodos` 的 for 循环内，为每个 `todo` 创建 `<select class="todo-priority" data-id="${todo.id}">`，三个 option 与添加表单一致，`value` 为 `todo.priority ?? "normal"` 的 option 设 `selected`。将该 select 插入到复选框与文案之间（或文案与删除按钮之间），避免误触编辑。下拉的 change 事件委托到 `#todo-list`。

**Step 2: 实现 handleChangePriority(id, newPriority, listEl)**

- 根据 id 找到项，设置 `item.priority = newPriority`。
- 使用 `sortedTodos(todos)` 得到新顺序，对结果 `forEach((t, i) => { t.order = i; })`。
- 调用 `saveTodosAndRender(listEl)`。

**Step 3: 在 listEl 上委托 change 事件**

若当前 listEl 只有 checkbox 的 change，可再在 change 里判断 `e.target.classList.contains("todo-priority")`，若是则 `handleChangePriority(e.target.dataset.id, e.target.value, listEl)`。

**Step 4: 手动验证**

修改某条优先级，确认列表立即重排并持久化；刷新或重启应用后顺序与优先级仍正确。

**Step 5: Commit**

```bash
git add desktop-todolist/src/main.js desktop-todolist/src/styles.css
git commit -m "feat(ui): 列表项展示并可修改优先级"
```
